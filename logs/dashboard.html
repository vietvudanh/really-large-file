All four variants share one codebase; the only difference is I/O strategy, and it costs a 2.4x spread (41.6s → 100.2s) on identical logic.

Agrees everywhere

Every run that reports it — 11 of the 20 — lands on the same names for record #432 and #43243: BECKER, FERDINAND F. and BAKER, JEFFREY MICHAEL. Months-per-donation also agrees everywhere it's computed: 66 distinct months.

SMITH vs. JOHN — a real algorithm split, not a bug

Go v1–v3, both Rust versions, Scala, and every Python run take everything before the first comma in column 8 as the counting key — which is the surname field in a "LAST, FIRST MIDDLE" record — so their most-common "first name" is actually the most common surname: SMITH. Go v9_marc and all four Java builds instead take the token right after ", ", which is the true first name, and land on JOHN — arguably the answer the task description actually asked for.

Same algorithm, slightly different count

Among the "surname" camp, Go and Rust report SMITH occurring 155,547 times; Scala and Python report 155,536 — an 11-record gap on ostensibly identical logic. Likewise Go v9_marc's JOHN count (563,412) is 7 higher than Java's (563,405). Almost certainly edge-case rows (missing commas, stray whitespace) handled slightly differently.

The one true outlier

Rust v0 is the single run of 20 that disagrees on the line count itself: 21,730,732 instead of the 21,730,731 every other implementation converges on. Rust v1 and v2, built after it, don't reproduce the discrepancy.

Does "pypy is fucking slow" hold up?

The project's own readme.md lessons-learnt section calls PyPy "fucking slow." Today's three head-to-head pairs tell a more mixed story.

Verdict: on the trivial line-counting case, PyPy wins convincingly — the opposite of the readme's claim. Once the workload does real per-line parsing (splitting columns, building dictionaries), the result flips per run: slower on python.v1, faster on python.v2. Net effect across the two full-pipeline scripts is close to a wash, well inside run-to-run noise. So: contradicted for the simple case, not clearly supported for the realistic one.

Then vs. now

2019 vs. 2026

Not an apples-to-apples comparison. The 2019 numbers in writeup.md, log.log, and bench.log ran against itcont_2018_20020411_20170529.txt — 1,999,820 lines, about a tenth the size of today's 21,730,731-line file — on unspecified/older desktop and laptop hardware (an i5 desktop and an i7-4790HQ laptop are named). Absolute seconds aren't comparable across a 10x-larger input and unknown, different machines. Every lineage got faster in throughput, not just in wall time on a bigger file — Rust roughly 18–26× more lines per second, Go 11–15×, Scala 14×, Python 21–43× depending which 2019 machine it's measured against. Some of that is newer hardware, some is six years of compiler and runtime improvement, and some is likely OS page-cache behavior on a file this size — the logs don't isolate which. What both eras agree on: SMITH was already the answer the "before-the-comma" lineage produced back in 2019, so that algorithmic choice predates this whole benchmark, not something introduced today.
