Processing a 4.3 GB campaign-finance file, twenty-two different ways

Five language stacks — Go, Rust, Scala, Python, Java — each solve the same four tasks against itcont.txt: count the lines, print the name at record #432 and #43243, tally donations per month, and find the most common first name. Twenty-two runs, one machine, one language at a time so no run steals CPU from another. The newest two implementations run so close to the SSD's raw read speed that the disk, not the code, is now the bottleneck.

Hardware & bottleneck analysis

With the v4/v3 generation, the interesting question stopped being "which language is faster" and became "what is the program actually waiting on." On this machine — Apple M2, 4 performance + 4 efficiency cores, 16 GB RAM — the answer is now: the SSD.

The disk is the ceiling

Raw sequential read measured with dd: 1.71 GB/s. Reading the 4.29 GB file therefore takes at least 2.51s no matter what the code does. Rust v3 (2.77s) and Go v4 (2.80s) run at ~90% of that ceiling — they are I/O-bound. The two languages tie because both are waiting on the same disk, not because they compute equally fast.

Rust wins the part that isn't the disk

Behind the identical wall clocks, Rust v3 burns 2.4s of user CPU to Go v4's 3.2s — about 30% less work for the same answers. On faster storage (or with the file cached), Rust would pull ahead; here the SSD hides the difference.

macOS trap: mmap serializes page faults

The first fast implementation used mmap and ran 6.5s pinned at ~1 core: macOS serializes concurrent page faults on a mapped file behind a per-VM-object lock, so eight threads faulting the same mapping queue up single-file. Switching to pread per worker (each thread reads its own 128 MB chunk) fixed it. On Linux, mmap would likely win instead — this result is OS-specific.

Every run pays for the full read

The 4.3 GB file does not stay in the 16 GB machine's page cache between runs — warm reruns time identically to cold ones. So these numbers all include the full disk read, and the 2.51s floor applies to every row in the ranking above.

Evolution of the bottleneck

The version history reads as a tour of successively removed constraints. Naive single thread (Go v1 17.7s, Rust v1 10.5s, Python ~28–31s): CPU-bound on per-line parsing and allocation. Concurrent with shared state (Go v2/v3 3.8–4.3s, Rust v2 6.6s): bound by the single reader goroutine/thread and per-line allocations feeding a mutex or channel. Share-nothing, zero-alloc (Go v4 2.80s, Rust v3 2.77s — pread work-stealing chunks, SIMD byte scanning, flat arrays and arena/Fx hash tables): bound by the SSD itself. Past this point, buying speed means buying hardware.

All four variants share one codebase; the only difference is I/O strategy, and it costs a 2.4x spread (41.6s → 100.2s) on identical logic.

Agrees everywhere

Every run that reports it — 13 of the 22 — lands on the same names for record #432 and #43243: BECKER, FERDINAND F. and BAKER, JEFFREY MICHAEL. Months-per-donation also agrees everywhere it's computed: 66 distinct months.

SMITH vs. JOHN — a real algorithm split, not a bug

Go v1–v3, both Rust versions, Scala, and every Python run take everything before the first comma in column 8 as the counting key — which is the surname field in a "LAST, FIRST MIDDLE" record — so their most-common "first name" is actually the most common surname: SMITH. Go v9_marc and all four Java builds instead take the token right after ", ", which is the true first name, and land on JOHN — arguably the answer the task description actually asked for.

Same algorithm, slightly different count

Among the "surname" camp, Go and Rust report SMITH occurring 155,547 times; Scala and Python report 155,536 — an 11-record gap on ostensibly identical logic. Likewise Go v9_marc's JOHN count (563,412) is 7 higher than Java's (563,405). Almost certainly edge-case rows (missing commas, stray whitespace) handled slightly differently.

The one true outlier

Rust v0 is the single run of 22 that disagrees on the line count itself: 21,730,732 instead of the 21,730,731 every other implementation converges on. Rust v1, v2, and v3, built after it, don't reproduce the discrepancy.

Then vs. now

2019 vs. 2026

Not an apples-to-apples comparison. The 2019 numbers in writeup.md, log.log, and bench.log ran against itcont_2018_20020411_20170529.txt — 1,999,820 lines, about a tenth the size of today's 21,730,731-line file — on unspecified/older desktop and laptop hardware (an i5 desktop and an i7-4790HQ laptop are named). Absolute seconds aren't comparable across a 10x-larger input and unknown, different machines. Every lineage got faster in throughput, not just in wall time on a bigger file — Rust roughly 18–26× more lines per second, Go 11–15×, Scala 14×, Python 21–43× depending which 2019 machine it's measured against. And the 2026 frontier has since moved past this table: the new Rust v3 and Go v4 builds (~2.8s, ~7.8M lines/s) run at ~90% of the SSD's raw read speed, roughly 40× the throughput of the best 2019 run. Some of that is newer hardware, some is six years of compiler and runtime improvement, and some is likely OS page-cache behavior on a file this size — the logs don't isolate which. What both eras agree on: SMITH was already the answer the "before-the-comma" lineage produced back in 2019, so that algorithmic choice predates this whole benchmark, not something introduced today.
